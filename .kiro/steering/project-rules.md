## 0. 設計思想の憲法（最重要）

### 非恣意性（Non-arbitrary）の原則
- シミュレーション中に、特定の成果（例: インターネット、言語統一、国家誕生など）を保証・誘導するイベントやルールを追加しない
- ルールの追加は「物理的・計算的に破綻するため最低限必要」な場合に限定し、追加理由を明文化する（**公理予算**）
- 観測指標（メトリクス）は世界を動かす目的関数として使わず、**ログ解析のみ**に使う
- スコア最大化ゲームにしない（すると合理世界に寄りすぎる）

### 局所性（Locality）と遅延（Delay）
- エージェントの知覚は局所に限定し、ノイズを含む
- 通信・移動・物流は距離に応じた遅延とコストを持つ
- 遠方の事象は、情報が到達しない限り意思決定に利用できない

### 有限性（Scarcity）と維持コスト
- エネルギー、時間、資源、空間、注意、記憶、インフラ耐久は有限とする
- 物資・情報・制度・インフラには維持コストと劣化（散逸）を必ず導入する
- 「無限コピー」「無料通信」「無料移動」を許さない

### 偶然（Stochasticity）と経路依存（Path dependence）
- 言語が分かれる、技術が生まれる、国家ができる、などは必然ではなく偶然の積み重ねとして扱う
- 乱数は必要（ただし再現性のためseed管理はする）

### 観測はするが、誘導しない
- 指標は「世界を動かす目的関数」ではなく「後から読むログ」
- 創発した現象を記録・分析するが、特定の方向に誘導しない

## 1. アイデア管理ルール

### 原文の保存
- 新しいアイデアが出た場合、該当するスペックディレクトリに `original-idea.md` として原文を保存する
- 日時を必ず記載する
- 保存先: `.kiro/specs/{feature-name}/original-idea.md`

### 共通思想の要約
- `.kiro/ideas/README.md` に共通思想・方向性の要約を記載・更新する
- 新しいスペックが追加されたら、関連スペック一覧を更新する

---

## 2. Git管理ルール

### コミット
- 変更は適宜コミットする
- コミットメッセージは変更内容を簡潔に記載する

### ブランチ
- mainブランチとしてプッシュする
- 作業はmainブランチで直接行う

### プッシュタイミング
- 重要な変更（スペック作成、設計完了、実装完了など）の後にプッシュする
- 細かい修正は複数まとめてプッシュしても良い

## 3. ディレクトリ構造

```
.kiro/
├── specs/                    # 各機能のスペック
│   └── {feature-name}/
│       ├── original-idea.md  # 原文と日時
│       ├── requirements.md   # 要件
│       ├── design.md         # 設計
│       └── tasks.md          # タスク
├── ideas/
│   └── README.md             # 共通思想・方向性の要約
├── reports/                  # 検証レポート・考察
│   ├── CONCLUSIONS.md        # 結論集（蓄積・更新）

└── steering/
    └── project-rules.md      # このファイル（プロジェクトルール）
```

## 4. レポート管理ルール

### 目的
- シミュレーション結果の検証、現世界との照合、気づき・考察を記録する
- 観測結果を蓄積し、後から参照できるようにする

### ファイル構成
- `CONCLUSIONS.md`: 結論集（蓄積・更新される単一ファイル）
- `YYYY-MM-DD-{topic}.md`: 個別の検証レポート

### CONCLUSIONS.md（結論集）
シミュレーションを重ねるたびに更新され、研ぎ澄まされていく結論をまとめる。

**構成**:
1. **確立された結論**: 複数の検証で支持された普遍的知見
2. **仮説（検証中）**: 観察されたが未検証の仮説
3. **棄却された仮説**: 検証の結果、支持されなかった仮説
4. **未解決の問い**: まだ答えのない問い
5. **更新履歴**: いつ何を更新したか

**更新ルール**:
- 新しいシミュレーションで結論が支持されたら、根拠レポートを追加
- 新しい発見があれば、仮説または結論として追加
- 仮説が十分に検証されたら「確立された結論」に昇格
- 仮説が否定されたら「棄却された仮説」に移動
- 結論は「研ぎ澄ます」ことを意識し、より本質的な表現に更新

### 個別レポート（YYYY-MM-DD-{topic}.md）
個々のシミュレーション・検証の詳細を記録する。

**記載内容**:
1. **日時**: レポート作成日
2. **概要**: 何を検証・観察したか
3. **検証結果**: データ、数値、比較表
4. **現世界との照合**: 類似する現象、理論との対応
5. **考察・気づき**: 発見、仮説、疑問
6. **今後の観察ポイント**: 次に調べたいこと

### 注意事項
- レポートは「観測結果」であり、シミュレーションを誘導するために使わない
- 恣意的な解釈を避け、データに基づいて記述する
- 仮説は仮説として明記する
- 結論集は「答え」ではなく「現時点での理解」として扱う

## 5. 公理予算ルール

新しい公理（ルール）を追加する際は、以下を満たすこと：

1. **必要性の証明**: 「これがないと物理的・計算的に破綻する」ことを明文化
2. **最小性**: より少ない公理で同じ効果が得られないか検討
3. **記録**: 追加理由と日時を記録
4. **レビュー**: 追加後、恣意性が入っていないか定期的にレビュー
